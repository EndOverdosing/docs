Vyla API aggregates six independent streaming providers. Each has different internal mechanisms, reliability profiles, and timeout characteristics. This page explains what's happening under the hood so you can build smarter applications.

---

## How Providers Are Used

On every `/api/movie` or `/api/tv` request:

1. All six providers are queried **simultaneously** with per-source random jitter to stagger upstream requests
2. Each provider has its own **timeout**, **retry count**, and **backoff delay**
3. Every returned URL is **verified** by fetching it and checking for `#EXTM3U`
4. Only URLs that pass verification appear in the response

The total response time is determined by the slowest provider that returns a verified URL.

## Provider Reference

### Source1

| Property | Value |
|----------|-------|
| Key | `source1` |
| Proxy param | `s1` |
| Timeout | 15s |
| Retries | 3 |
| Jitter | 400ms |

Uses a two-stage AES decryption flow. First fetches an API key from `core.source1.com`, derives a symmetric key via SHA-256, then iterates up to 14 server slots decrypting each candidate link. Verified links are probed with `HEAD` before being returned. Generally reliable and fast.

---

### Source2

| Property | Value |
|----------|-------|
| Key | `source2` |
| Proxy param | `s2` |
| Timeout | 20s |
| Retries | 3 |
| Jitter | 600ms |

Uses a custom Base64 alphabet for payload decryption, with six internal server backends tried sequentially. Each backend has a different response schema. Source2 typically has fast response times and good availability.

---

### Source3

| Property | Value |
|----------|-------|
| Key | `source3` |
| Proxy param | `s3` |
| Timeout | 25s |
| Retries | 2 |
| Jitter | 700ms |

Uses a three-hop HTML scraping chain: load embed page → extract iframe URL → extract player URL → extract `file:` field with domain placeholders. The longest timeout of any provider at 25s. Availability is inconsistent — check `/api/health` before relying on it.

---

### Source4

| Property | Value |
|----------|-------|
| Key | `source4` |
| Proxy param | `s4` |
| Timeout | 20s |
| Retries | 3 |
| Jitter | 800ms |

Encrypts TMDB ID using AES-CBC with a fixed passphrase before fetching stream URL. Includes an additional `isBlocked()` check that fetches final URL and rejects it if response is HTML (a common blocking pattern). Hit-or-miss depending on day.

---

### Source5

| Property | Value |
|----------|-------|
| Key | `source5` |
| Proxy param | `s5` |
| Timeout | 20s |
| Retries | 3 |
| Jitter | 900ms |

Fetches encrypted blobs from six internal Source5 server backends concurrently, then decrypts them via an external API at `decrypt.example.com`. Results are cached in memory to avoid redundant decrypt calls. Generally reliable with moderate response times.

---

### Source6

| Property | Value |
|----------|-------|
| Key | `source6` |
| Proxy param | `s6` |
| Timeout | 15s |
| Retries | 2 |
| Jitter | 500ms |

The simplest provider. Constructs a direct M3U8 URL (`source6.com/v1/stream/master/...`) and validates it with a HEAD request. No encryption or scraping. Fast and deterministic when available.

## Reliability by Provider

Based on typical behavior — check `/api/health` for live status.

| Provider | Speed | Availability | Notes |
|----------|-------|--------------|-------|
| Source1 | Fast | High | Strong first choice |
| Source2 | Fast | High | Consistent |
| Source6 | Very fast | Medium | Simplest, lowest overhead |
| Source5 | Medium | Medium-High | External decrypt dependency |
| Source4 | Medium | Medium | Blocked-URL check adds latency |
| Source3 | Slow | Low | Last resort; high timeout |

## Provider Timeouts and Your Response Time

The total response time for `/api/movie` or `/api/tv` is gated by:

- The **slowest provider** that returns a verifiable URL
- Plus the **verification HEAD request** for each candidate

In the best case (all providers fast, first one verified), you'll see ~2–3s. In the worst case (only Source3 works), you're looking at 25s+ before the response arrives.

**Recommended:** always show a loading state and implement a timeout on your side as well.

## Caching

Results are cached in a Cloudflare Worker in-memory `Map` with a **5-minute TTL**, keyed by `source + tmdb_id + season + episode`. Cache hits skip the provider fetch entirely and return near-instantly.

<Warning>
  The cache is per-Worker-isolate. Cloudflare Workers can run across many isolates globally, so cache hit rates in production are not 100%. Cached responses are best-effort, not guaranteed.
</Warning>

## The HLS Proxy

All provider stream URLs are wrapped in the Cloudflare proxy:

```
/api?url=<encoded_stream_url>&<proxy_param>=1
```

When your player fetches this path, the Worker:

1. Fetches the upstream M3U8 with the correct `Referer` and `Origin` headers for that provider
2. Rewrites all segment URLs and encryption key URIs in the playlist to route through the same proxy
3. Returns the rewritten playlist with `Access-Control-Allow-Origin: *`

This means **your player never talks to the provider directly** — all HLS traffic flows through Cloudflare, keeping CORS and rate-limiting issues fully contained.
